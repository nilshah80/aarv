## [0.9.1] - 2026-06-06

### Fixed

- **`plugins/hmacauth` signing now works across redirects on Go 1.22.** Go 1.22's `net/http.Client` populates the redirect-target request's `Body` from the original request's `GetBody()` but does not propagate `GetBody` itself to the new request (fixed upstream in Go 1.23). On Go 1.22, `SigningTransport.CheckRedirect` and `ResignOnRedirect` therefore saw `req.Body` set but `req.GetBody == nil` and returned `hmacauth: req.GetBody is nil â€” request body is not replayable`, breaking the existing `TestSigningTransportFollowsRedirect` and `TestResignOnRedirectFunction` under the Go 1.22 CI matrix. The fix is internal to `readReplayableBody`, which now falls back to reading `req.Body` directly when `GetBody` is nil AND `req.ContentLength > 0`; the positive-length gate preserves the existing rejection of truly unreplayable streams (`io.Pipe` and friends have `ContentLength == 0` or `-1` â€” reading those would block or pull arbitrary upstream data). After the fallback read, `req.Body` is rebuffered via `io.NopCloser(bytes.NewReader(b))` and `req.GetBody` is populated so the request stays replayable for the redirect target. Both `SigningTransport.CheckRedirect` and `Transport.RoundTrip` benefit transparently because they already called `readReplayableBody`; `ResignOnRedirect` was restructured to delegate to `readReplayableBody` instead of its own GetBody-only path, inheriting the same fallback. **Sign() itself does NOT get the fallback** â€” Sign's body-resolution rules (`resolveSignBody`) are unchanged and still reject `body=nil + req.Body!=nil + req.GetBody==nil`; callers using the lower-level `Sign()` directly with that shape must pass body bytes explicitly. Regression test: `TestReadReplayableBody_FallsBackToBodyWhenGetBodyNil`. No public API change; consumers on Go 1.23+ are unaffected.

### Internal

- `plugins/hmacauth/signer.go` doc-comment + error-message clarifications: `Sign()`'s godoc gains a "Sign-vs-Transport asymmetry" paragraph spelling out that Sign remains GetBody-only while the transport helpers got the Go-1.22 fallback. The corresponding `resolveSignBody` rejection error now lists the alternative entry points (body parameter / typed reader / `Transport` / `SigningTransport.CheckRedirect` / `ResignOnRedirect`). `Transport` / `SigningTransport.CheckRedirect` / `ResignOnRedirect` / `readReplayableBody` docstrings updated to describe the fallback and (in `ResignOnRedirect`'s case) correctly state that streaming bodies cause the redirect to abort client-side rather than reach the destination. Behavior change is limited to the readReplayableBody fallback above; the error-string text is observable for log-grep consumers but not part of Go's public API surface.
- `gofmt -s` sweep across three files where alignment whitespace had drifted: `plugins/hmacauth-otel/observer_test.go`, `plugins/hmacauth/observer.go`, `plugins/ratelimit-redis/coverage_test.go`. No behavior change.

### Not re-tagged

Submodule pins stay at root `v0.9.0` â€” none of `plugins/{prometheus,otel,sanitize,ratelimit-redis,hmacauth-otel}` consume code paths affected by this fix, so Stage 2 retagging would be churn. Consumers who pinned root at `@v0.9.0` can move to `@v0.9.1` with no API change required.

## [0.9.0] - 2026-06-05

### Breaking Changes

- **`aarv.RegisterNativeMiddleware(m, fn)` return type changed: `aarv.Middleware` â†’ `aarv.NativeMiddleware`.** Every plugin's `New(...)` constructor that returns the result of `RegisterNativeMiddleware` now returns `NativeMiddleware` too. See the per-plugin list under [Migration](#migration).
- **`aarv.WrapMiddleware(fn)` return type changed: `aarv.Middleware` â†’ `aarv.NativeMiddleware`.** Helper functions like `func myMW() aarv.Middleware { return aarv.WrapMiddleware(fn) }` no longer compile; change the return type word to `aarv.NativeMiddleware`.
- **`aarv.SkipPaths(paths []string, mw any) aarv.NativeMiddleware`** â€” was `(mw Middleware) Middleware`. The `mw` parameter widened to `any` (accepts `Middleware`, `NativeMiddleware`, or `func(http.Handler) http.Handler`); panics on nil or unsupported type via `coerceSlot`.
- **`App.Use`, `RouteGroup.Use`, `PluginContext.Use`, `WithRouteMiddleware` variadic widened from `...Middleware` to `...any`.** Source-compatible for the overwhelming majority of call sites (`Middleware` satisfies `any`). The **spread pattern `app.Use(mws...)` where `mws` is `[]aarv.Middleware` no longer compiles** â€” rewrite as `mws := []any{...}` or call `app.Use(mw)` in a loop. Misuse (nil, unsupported types) panics at registration with a clear message.
- **`nativeMiddlewareRegistry` and `nativeMiddlewareFunc` deleted entirely** (private symbols; consumers were not expected to use them).
- **`pprof.Config.AuthMiddleware` stays typed `aarv.Middleware`** (the field is invoked only on the stdlib path at [`pprof.go:98`](plugins/pprof/pprof.go)). Consumers with a `NativeMiddleware` (e.g., `jwtMW := jwt.New(...)`) extract `.Stdlib` when assigning to this field: `cfg.AuthMiddleware = jwtMW.Stdlib`.

### Fixed

- **Multi-instance wrapper collisions on the now-deleted code-pointer registry.** Before v0.9.0, two instances of the same wrapper-style middleware (e.g., two `aarv.SkipPaths` calls, two `aarv.WrapMiddleware` calls) silently overwrote each other's native-fn registration because `reflect.ValueOf(m).Pointer()` returns the same code pointer for closures of the same source function. The new `NativeMiddleware`-as-value design eliminates the collision at the source â€” each value carries its own identity.
- **`aarv.SkipPaths` preserves the native fast path** under multiple distinct instances (regression test `TestSkipPaths_PreservesNativeFastPathAcrossInstances` locks this in).

### Added

- **`aarv.NativeMiddleware{Stdlib, Native}` value type** â€” bundles a stdlib `Middleware` with its aarv-native `MiddlewareFunc` counterpart. Each value carries its own identity. Returned by `RegisterNativeMiddleware`, `WrapMiddleware`, `SkipPaths`, every plugin's `New(...)`, and the built-in `aarv.Recovery()` / `aarv.Logger()`.
- **`aarv.StatusRecorder`** â€” public `http.ResponseWriter` wrapper that captures the final HTTP status code and total body bytes written. Constructor `aarv.NewStatusRecorder(w)` returns an unpooled, simple-lifecycle value. `Reset(w)` is provided for callers that want to pool `*StatusRecorder` privately. Implements `Unwrap() http.ResponseWriter` so `http.ResponseController` walks through to the underlying writer's `Flusher` / `Hijacker` / `Pusher`. Already adopted by `plugins/logger`.
- **`plugins/hmacauth` `Config.Observer` hook** with `Outcome` / `Event` types. Fires once per verification attempt (after Skipper bypass; gated on `Observer != nil` for zero overhead when unset). Outcome enum: `OutcomeOK`, `OutcomeUnauthorized`, `OutcomeClockSkew`, `OutcomeSignatureInvalid`, `OutcomeReplayDetected`, `OutcomeBodyTooLarge`. Nonce-store transport errors report `OutcomeUnauthorized` not `OutcomeReplayDetected` so Redis outages don't look like flood of replay attempts; body-read transport errors report `OutcomeUnauthorized` not `OutcomeBodyTooLarge`.
- **`plugins/hmacauth-otel` companion module** (`v0.1.0`) â€” OpenTelemetry adapter that converts each `hmacauth.Event` into an `auth.HMAC.verify` span with the canonical attribute schema (`auth.client_id`, `auth.outcome`, `auth.response_status`, `auth.skew_seconds`). The span is back-dated by `Event.Duration` so its recorded interval matches the actual verification window. Span status is set to `codes.Error` on any outcome other than `OutcomeOK`. Lives in its own go module because root must stay zero-dep â€” OTel can't import into root-module plugins.
- **`plugins/openapi` `Config.Tags`** â€” declarative tag filter. When non-empty, only routes carrying at least one of the listed tags (set via `aarv.WithTags(...)`) are included in the spec. Ignored when `Include` is set; applied before `Exclude`.
- **`plugins/prometheus` `SubMillisecondBuckets`** histogram preset â€” `100Âµs..5s`, sized for low-latency services whose p50 falls below the `DefaultBuckets` 1ms first bucket.
- **`docs/middleware.md`**: new "Wrapping a middleware to add observability" recipe covering the stdlib-only / native-only / both-paths patterns plus `c.SetContext` / `c.BindRequest` / `RegisterNativeMiddleware` usage.

### Changed

- **`plugins/otel` HTTP semantic-convention attributes** migrated to OpenTelemetry semconv v1.37.0 (aligned with the plugin's existing OTel pin at v1.43.0). Spans emit modern keys (`http.request.method`, `url.path`, `http.route`, `http.response.status_code`, `user_agent.original`, `client.address`, `network.protocol.version`) alongside the legacy keys (`http.method`, `http.target`, `http.status_code`, `http.user_agent`, `net.peer.ip`) for one transitional minor release. Metric attribute sets use `http.route` only â€” `url.path` is deliberately NOT emitted on metrics (per-URL labels are a TSDB cardinality bomb; locked in by `TestSemconv_MetricsUseRouteNotURLPath`). Legacy `http.target` preserves its pre-migration shape (route pattern when matched, raw path otherwise) â€” do NOT confuse it with modern `url.path`. **Legacy keys deprecated in v0.9.0, removed in v0.10.0** (no earlier than 4 weeks after v0.9.0 ships). Migrate any TraceQL / Datadog / Honeycomb / Grafana queries to the modern keys before then. `request_id` keeps its name (aarv-specific, not a semconv rename).
- **`plugins/logger`**: internal `responseWriter` replaced with `aarv.StatusRecorder` (still pooled privately via `sync.Pool`). No behavior change.

### Migration

- **Plugin constructor return types changed.** `app.Use(plugin.New(...))` works unchanged. `var m aarv.Middleware = plugin.New(...)` no longer compiles â€” use `var m aarv.NativeMiddleware = plugin.New(...)` or `var m aarv.Middleware = plugin.New(...).Stdlib`.
- **Helper functions returning `aarv.Middleware` via `aarv.WrapMiddleware`** must change their return type to `aarv.NativeMiddleware`.
- **`app.Use(mws...)` spread** with `mws []aarv.Middleware` no longer compiles. Rewrite as `mws := []any{...}` or loop.
- **`pprof.Config.AuthMiddleware = jwtMW` where `jwtMW` is `NativeMiddleware`** no longer compiles. Use `cfg.AuthMiddleware = jwtMW.Stdlib`.
- **Tuple constructors** (`encrypt.New`, `ratelimit.NewWithCleanup`) return `(aarv.NativeMiddleware, ...)` â€” only the type word changes. **Watch the error-path return**: `aarv.NativeMiddleware` is a struct, not a pointer, so the zero-value return is `aarv.NativeMiddleware{}` not `nil`:
  ```go

  return nil, ErrInvalidKey
  // After (v0.9.x):
  return aarv.NativeMiddleware{}, ErrInvalidKey
  ```

Per-plugin migration scope (`plugins/timeout.New(d)` and `pprof.Config.AuthMiddleware` are the only exceptions that stay `aarv.Middleware`):

- In-root constructors changing to `NativeMiddleware`: `apikey.New`, `basicauth.New`, `bearer.New`, `bodylimit.New` + `bodylimit.NewWithResponse`, `compress.New`, `cors.New`, `csrf.New`, `encrypt.New` + `encrypt.MustNew`, `etag.New`, `health.New`, `hmacauth.New`, `idempotency.New`, `ipfilter.New`, `jwt.New`, `logger.New`, `pprof.New`, `ratelimit.New` + `ratelimit.NewWithCleanup`, `rbac.(*Authorizer).RequireRoles` + `RequireAnyRole`, `recover.New`, `requestid.New`, `secure.New`, `session.New` + `session.NewCookie`, `static.New`, `throttle.New`, `timeout.Context`, `verboselog.New`.
- Submodule constructors changing: `otel.New`, `prometheus.New`, `ratelimit-redis.New`, `sanitize.New`.
- Submodules unaffected by the constructor migration (pin bump only): `autocert`, `h2c`, `hmacauth-redis`, `idempotency-redis`, `openapi`, `openapi-ui`.

## [0.8.0] - 2026-05-05

### Added

- `plugins/session` (root module, stdlib-only): cookie-tracked HTTP session middleware. Includes the following review-driven hardenings landed pre-merge so the initial release ships clean:
  - **Load-error fallback routes through `backend.freshSession`.** `freshSession(cfg *normalized) (*Session, error)` was promoted to the internal `sessionBackend` interface so the native and stdlib middleware paths' "ErrorHandler returned nil; mint a fresh session and proceed" branch goes through backend-specific framing rather than handcrafting a session via `generateID + newSession`. This is what kept the IDLength-honor bug latent for so long: the duplicated handcraft bypassed `cfg.idLength`. It also matters going forward â€” any future backend (CookieStore, session-redis, session-sql) that needs special semantics for "no usable inbound state" gets it for free here. Covered by `TestLoadErrorFallbackUsesBackendFreshSession` (native path) and `TestLoadErrorFallbackStdlibPathUsesBackendFreshSession` (stdlib path) using a tracking-backend wrapper that asserts `freshSession` is invoked exactly once on the fallback.
  - **CookieStore AEAD bound to cookie name.** AES-256-GCM Seal/Open now pass `[]byte("aarv/session:" + cfg.cookieName)` as additional authenticated data, so a value sealed under cookie name A cannot be replayed as cookie name B even when the same master key is reused for both. The `"aarv/session:"` prefix is a domain-separation tag so a non-session AES-GCM use of the same key cannot collide by coincidence with a short session AAD value. Previously both Seal and Open passed nil AAD. Covered by `TestCookieStoreCrossNameReplayRejected`.
  - **Destroy after Regenerate clears both server IDs.** When a handler regenerates the session and then destroys it in the same request, `storeBackend.destroy` now deletes both `sess.id` and `sess.oldID` (when distinct) so the pre-regen ID does not survive until TTL. The expiration cookie is still always written, even if both Delete calls fail; SaveErrorHandler receives the most recent error. Covered by `TestDestroyAfterRegenerateDeletesBothIDs` and `TestDestroyAfterRegenerateStillExpiresCookie`.
  - **Stdlib middleware path syncs context with upstream-mutated request.** When invoked via the stdlib path, the session middleware now calls `c.BindRequest(r)` on entry (when `*aarv.Context` is reachable) so the skipper, the backend load, and the downstream handler all observe upstream URL rewrites, header injection, body decompression, prefix stripping, and `r.WithContext(...)` mutations. Mirrors the `plugins/rbac` and `plugins/bearer` adoption pattern. Covered by `TestStdlibPathSyncsContextWithUpstreamRewrite`.
  - **`IDLength` validated at construction.** `New` and `NewCookie` panic on `IDLength` in the `(0, MinIDLength)` range so a footgun-low value (e.g. `IDLength: 1` producing 2-character IDs) cannot ship to production. Zero takes the default (32); `>= MinIDLength` (16) is accepted. `MinIDLength` mirrors the CSRF plugin's TokenLength minimum. Covered by `TestNewPanicsOnFootgunIDLength`, `TestNewCookiePanicsOnFootgunIDLength`, and `TestNewAcceptsZeroAndAtLeastMinIDLength`.
  - **Session-fixation defense.** `storeBackend.load` mints a fresh random ID when the client supplies a cookie whose ID is missing from the store (forged, expired, or rotated key). The previous "reuse the supplied id so the client's cookie remains valid" path let an attacker plant `_session=known-id` and then have the victim's login `Save` persist under that attacker-chosen ID â€” classic session fixation. Covered by `TestStoreBackendLoadMissingEntryMintsFreshID` (unit) and `TestNoFixationOnLoginAfterPlantedCookie` (end-to-end through the App).
  - **Logout reliability under backend failure.** `storeBackend.destroy` now writes the expiration `Set-Cookie` (Max-Age=-1) **regardless of whether `Store.Delete` succeeded**, then returns the Delete error for `SaveErrorHandler` reporting. The previous early-return left the browser holding a still-valid cookie when the backend was unhealthy. Covered by `TestDestroyExpiresCookieEvenWhenDeleteFails`.
  - **`Config.IDLength` is honored.** Previously the field existed but every generation site hardcoded `defaultIDLen` (32). The configured length is now stored on `normalized.idLength` and on `Session.idLen` so both fresh-session creation and `Regenerate` use it. Cookie store `freshSession` honors it too. Covered by `TestIDLengthHonoredOnFreshSession`, `TestIDLengthHonoredOnRegenerate`, and `TestIDLengthHonoredOnCookieStoreFreshSession`.
  - **`sessionWriter` conditionally forwards `http.Hijacker` and `http.Pusher`.** The previous wrapper exposed only `Flush` and `Unwrap`, so direct type-assertions for Hijacker (gorilla/websocket, nhooyr/websocket upgrade paths) or Pusher (HTTP/2 server push) silently failed even when the underlying writer supported them. `wrapWriter` now returns one of four variants picked from the underlying writer's capabilities â€” `*sessionWriter` (plain), `swHijacker`, `swPusher`, or `swHijackerPusher` â€” so the assertion succeeds when (and only when) it would succeed on the underlying writer. `Hijack` calls `commit()` first so any pending `Set-Cookie` lands in `w.Header()` before the upgrader serializes the WebSocket handshake response. `Push` does not call `commit()` â€” it initiates a separate request/response stream and does not flush headers on the original response. Five tests cover the variant matrix (Hijacker only, Pusher only, both, neither, plain).
  - **CookieStore Session.ID instability documented.** Under CookieStore the ID is the encrypted cookie value, so a fresh AES-GCM nonce per encryption means the ID changes on every save. `Session.ID()` godoc, `CookieStore` type godoc, and `doc.go` now spell this out and direct callers to use a stable application-level identifier (e.g. user ID stored via `Set`) for cross-request correlation, log keys, or external joins.

  Two backends: `MemoryStore` (default for dev / single-node, with optional janitor goroutine and `NewMemoryStoreWithJanitor` shutdown stop-func) and `CookieStore` (stateless AES-256-GCM-encrypted cookie payload, no server state). `Store` is the public server-backend interface; `CookieStore` intentionally does NOT satisfy it (request/response-scoped, fundamentally different load/save shape) and is wired via the separate `NewCookie(CookieConfig)` constructor so misuse is prevented at compile time. Optional `plugins/session-redis` and `plugins/session-sql` sub-modules are deferred for follow-up work.
  - **Cookie commit timing.** Session persistence is driven by a `sessionWriter` that wraps the response writer and runs the commit closure on the FIRST `WriteHeader` / `Write` / `Flush`, plus a post-handler fallback `commit()` for handlers that returned without writing. This means cookies land in the response headers even when the handler streams via `c.JSON` / `c.Blob` / `c.Text` (which call `WriteHeader` inline at handler time, not via `OnSend` hooks). The wrapper implements `Unwrap() http.ResponseWriter` (matches the convention used by `idempotency`, `prometheus`, `otel`) so `http.ResponseController` walks through to the underlying Flusher / Hijacker / SetReadDeadline. `c.SetResponse(orig)` is restored via `defer` around `next(c)` so panic / recovery paths cannot leave the framework Context pointing at the wrapper.
  - **Lazy persistence.** A newly-issued session that the handler never wrote to is NOT saved â€” `shouldSave = dirty || len(consumed) > 0 || regenerated`. First-touch health checks emit zero `Set-Cookie` headers and incur zero `Store.Save`. `dirty` is flipped by `Set` / `Delete` (on a present key) / `Flash` / `Regenerate` / `CSRFToken` first issuance; `consumed` is appended-to by `ConsumeFlash` so a save fires to drop the consumed key from the next request's `Stored.Flash`; `regenerated` is its own flag so multiple `Regenerate` calls in one request collapse to one old-ID delete and the original pre-regen ID is the one removed.
  - **Config defaults that are actually expressible.** `MaxAge time.Duration` zero â†’ `DefaultMaxAge` (24h); negative (or the `SessionMaxAge` sentinel) â†’ browser-session cookie (no Max-Age / Expires attributes; server-side TTL still uses `DefaultMaxAge` so the store entry doesn't outlive the browser). `Secure` and `HttpOnly` use the `Disable*` form (`DisableSecure`, `DisableHTTPOnly`) so their secure-by-default state is the zero value. `CookieSameSite` zero â†’ `http.SameSiteLaxMode`. `CookieName` zero â†’ `_session`. `CookiePath` zero â†’ `/`.
  - **Two-tier error handling.** `ErrorHandler func(*aarv.Context, error) error` runs for load-time errors (Store.Get failure) BEFORE the handler executes, so it can produce a controlled response (return non-nil to stop). Tampered cookies are NOT errors â€” backends translate them to `(nil, nil)` so the request continues with a fresh session, never reaching `ErrorHandler`. `SaveErrorHandler func(*aarv.Context, error)` is invoked for commit-time errors (Store.Save / Store.Delete / cookie encode failure) â€” side-effect only, since by the time it fires the response headers may already be flushed; default behavior is to log via the request-scoped `c.Logger()` (or `Config.Logger` / `slog.Default()` when `c == nil` on stdlib mounts) and swallow.
  - **Map-aliasing safety.** `MemoryStore` deep-copies `Stored.Data` and `Stored.Flash` on both `Save` (in) and `Get` (out) so concurrent requests for the same session ID can't share live map instances. `cloneStored` uses `maps.Clone` (shallow at the value level â€” values stored in a session must be treated as immutable once `Set`, documented in package docs). `-race` test exercises N concurrent writers + readers against a single key.
  - **CSRF storage is single-source.** `Stored.CSRF` is the only home for the per-session CSRF token; `_csrf` is NOT a reserved user key (a handler `Get("_csrf")` returns `(nil, false)`). `Session.CSRFToken()` lazy-issues, marks dirty so it persists, and is stable across calls within one request and across save/load round-trips. The CSRF plugin (10.3) will read it via `session.From(c).CSRFToken()` for session-bound tokens.
  - **Internal `sessionBackend` interface** takes `(c *aarv.Context, r/w, sess, cfg)` explicitly so the stdlib middleware path works correctly when `c == nil` (non-aarv mount). Both backends use `http.SetCookie(w, ...)` directly so writes flow through the wrapped writer regardless of which path is active.
  - **CookieStore value semantics.** `Stored` is JSON-marshaled into the encrypted cookie payload. Session values must be JSON-compatible; `int` decodes as `float64` (JSON has no integer type), `time.Time` decodes as an RFC3339 string, byte slices encode as base64. A 3.5 KiB encoded-payload guard at Save time stays safely under the ~4 KiB browser cookie limit; oversize payloads surface as a `SaveErrorHandler` call carrying `ErrCookiePayloadTooLarge`. Documented in `doc.go`.
  - Helpers: `session.From(c) (*Session, bool)` and `session.MustFrom(c) *Session` (panics if the middleware did not run for this route â€” surfaces missing wiring during development rather than as a confusing nil-deref later). Hardcoded `sessionContextKey = "session"` so `From` always finds it (mirrors the apikey / jwt / csrf convention).
  - Both native (`MiddlewareFunc`) and stdlib (`Middleware`) paths are registered via `aarv.RegisterNativeMiddleware` so middleware chains stay on the framework's native fast path when paired with other native-aware plugins. 95.4% test coverage (`-race`-clean); the remaining gap is defensive AES-key edge branches that cannot fire with a valid 32-byte key. Tests cover: cookie commit before `c.JSON`-style WriteHeader (the "would otherwise be a bug" case), clean-read no-cookie no-Save assertion via instrumented store call counts, post-handler commit fallback for no-write handlers, Regenerate `Delete(oldID)`, Destroy `Delete(currentID)` plus `MaxAge=-1` cookie, flash-consume forces save, skipper bypass, default-secure cookie attributes, SSE-style mid-stream `Flush` cookie commit, `Unwrap` chain via `http.NewResponseController`, tampered/expired cookie yields fresh session, stdlib-fallback path via stdlib-only sibling middleware, plus the five review-driven hardening tests called out at the top of this entry.
- Core: `(*Context).BindRequest(r *http.Request) *http.Request` â€” re-binds c to the supplied request, attaching the framework's context marker so `aarv.FromRequest(r)` succeeds and updating `c.req` so `c.Path` / `c.Method` / `c.Header` / `c.BodyReader` / `c.Context` all reflect r's URL, headers, body, and context. Use this from a stdlib `http.Handler` middleware that received a request mutated by an upstream middleware (URL rewrites, header injection, body decompression, prefix stripping, etc.). `SetContext` only swaps the context.Context on the framework's original `c.req`; `BindRequest` swaps the entire request, preserving every upstream mutation while keeping the c-recoverable contract intact. Request-derived caches on c are invalidated to match the swap: `c.query` is cleared, `c.bodyRead` is reset, and `c.bodyCache` is truncated (small-buffer reuse, large-buffer drop) so subsequent `c.Query()` / `c.Body()` reads come from the rebound request rather than returning stale values from the previous one â€” covered by `TestBindRequest_InvalidatesRequestDerivedCaches` and `TestBindRequest_ReusesBodyCacheBuffer`. Returns the request the caller must forward to next.ServeHTTP. Under `WithRequestContextBridge(false)` the previous request's pointer-keyed registry mapping is removed; under bridge mode (the default) the returned request carries an in-context marker so `FromRequest` still finds c via the context.Context chain. Adopted by `plugins/rbac` and `plugins/bearer` on their stdlib middleware paths so upstream `r.Clone()` mutations no longer get silently discarded.
- `plugins/problem` (root module, stdlib-only): RFC 7807 / RFC 9457 Problem Details for HTTP APIs. `Details` struct with the five standard members (Type / Title / Status / Detail / Instance) plus an `Extensions map[string]any` flattened into the top-level JSON object on marshal per RFC 7807 Â§3.2. **Reserved keys (`type`/`title`/`status`/`detail`/`instance`) are stripped from `Extensions` during the flatten copy** â€” even when the corresponding standard struct field is the zero value â€” so an extension can never inject a forged standard member, regardless of which fields the caller populated. `MarshalJSON` is on a value receiver so `json.Marshal(Details{...})` (value) and `json.Marshal(&Details{...})` (pointer) both flow through the custom marshaler â€” a pointer-receiver design would silently bypass it for the value form and emit Go field names. `Handler(Config) aarv.ErrorHandler` is the primary entry point â€” wire via `aarv.WithErrorHandler(problem.Handler(...))`. Conversion rules: `*aarv.AppError` â†’ status / Code â†’ Title / Message+Detail â†’ combined Detail; `*aarv.ValidationErrors` â†’ 422 with the per-field failures under the `errors` extension (matching `aarv.ValidationError` shape so clients can decode it directly); anything else â†’ 500 with a static `"Internal server error"` body â€” the original `err.Error()` is NEVER written to the wire (it may carry DB connection strings or stack traces) and the `OnInternal` callback receives the unmasked error for logging. **`OnInternal` also fires for `*aarv.AppError` 5xx responses that carry a non-nil `Internal()` value** (the `aarv.ErrInternal(downstream)` shape) and is passed the unwrapped `Internal()` err â€” matching the framework's default-handler `slog.Error` parity for users who switch to problem+json. Plain `*aarv.AppError` 5xx (e.g. `ErrServiceUnavailable("draining")`) still does NOT fire `OnInternal` since the caller chose that status to communicate state, not to signal a fault. Type resolution: `Config.TypeForCode[appErr.Code()]` â†’ `Config.Type` â†’ `DefaultType` ("about:blank"). `Config.Instance(c) string` populates per-occurrence URIs (typical: `c.Path()`). `request_id` extension auto-added when the framework's request-id middleware has populated `c.RequestID()`. Content-Type fixed at `application/problem+json` (`problem.ContentType` constant). `FromError(err)` and `ValidationProblem(errs)` are exposed for callers that need the structured form before serialization. Per-route opt-in/out is achieved by installing or omitting the handler globally; route-level toggling is intentionally out of scope (the handler runs after middleware returns and does not know which route was hit) and the package godoc points readers at a wrap-and-switch workaround. Defense-in-depth `marshalOrFallback` traps a non-marshalable extension value (channel, func, complex) and emits a hardcoded minimal conforming document. 100% test coverage; race-clean.

## [0.7.0] - 2026-05-02

### Added â€” Phase 10 Security Plugins
- IP filter plugin (`plugins/ipfilter`, root module, stdlib-only): allowlist or denylist filtering against a set of CIDRs (or bare IPs auto-converted to /32 / /128). Default source IP is `(*aarv.Context).RealIP()`; overridable via `IPFunc` for proxy fronts. Invalid CIDRs panic in `New` (parity with jwt). Empty CIDRs in `ModeAllowlist` panics â€” an empty allowlist would block all traffic, almost always a misconfiguration. Empty/unparseable source IP fails closed in allowlist mode and fails open in denylist mode (documented).
- Throttle plugin (`plugins/throttle`, root module, stdlib-only): bounds in-flight request count via a `chan struct{}` semaphore with optional bounded queue. Queue token is released as soon as the goroutine acquires a slot or its wait times out â€” never held for the duration of the handler â€” so queue depth is exactly `QueueSize` regardless of handler latency. Slot release is deferred so handler errors and panics don't leak slots; release fires before the panic propagates so Recovery middleware behavior is unchanged. Returns 503 (configurable) on queue full or wait timeout.
- Rate limit plugin (`plugins/ratelimit`, root module, stdlib-only): token-bucket (default) and sliding-window algorithms over a sharded keyed store. Default key is `(*aarv.Context).RealIP()`; custom `KeyFunc` for per-user / per-API-key limits. Headers `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset` always set; `Retry-After` set on 429. **No goroutines started in `New(cfg)`**: stale entries are pruned in-line via a deterministic `atomic.Uint64` counter â€” every Nth call sweeps one shard, cycling through all shards over time. `NewWithCleanup(cfg)` returns a periodic janitor goroutine plus a stop function, wired via `app.OnShutdown`. Sweep counter advances on every limiter check (admitted + denied), so denial pressure also drives cleanup.
- CSRF plugin (`plugins/csrf`, root module, stdlib-only): double-submit cookie pattern with `crypto/subtle` constant-time comparison over base64-decoded tokens. Default `CookieHTTPOnly: false` for SPA/API ergonomics (the classic double-submit pattern requires JS to copy the cookie into the request header); server-rendered apps can set `CookieHTTPOnly: true` and read `csrf.Token(c)` to inject into HTML/meta/form. `SafeMethods` follows nil-vs-empty semantics: `nil` â†’ defaults `{GET, HEAD, OPTIONS, TRACE}`; `[]string{}` â†’ every method requires a token. Optional `FormField` fallback for non-AJAX forms.
- Sanitize plugin (`plugins/sanitize`, **separate submodule** â€” pulls `golang.org/x/text/unicode/norm`, root stays zero-dep): JSON request-body sanitizer. Recursively walks decoded JSON, applying stdlib HTML stripping, NFC Unicode normalization, and any caller-supplied `SanitizerFunc`s, then replaces `r.Body`. Field allowlist/blocklist, configurable Content-Types, body-size cap with 413 response on overflow. Invalid JSON passes through unchanged â€” sanitizer is not a body validator.
- Idempotency plugin (`plugins/idempotency`, root module, stdlib-only): RFC-aligned `Idempotency-Key` middleware. First request locks the key, captures the response (status + headers + body), saves under the key with TTL, and returns it. Retries replay verbatim with `Idempotency-Replayed: true` response header. Capture writer has an explicit overflow state machine: under-cap responses never write to the underlying writer until the middleware commits them after Save (so Save failure can decline to commit); over-cap responses transition to passthrough mid-stream, flush captured headers + buffered prefix with `Idempotency-Cached: false; reason=size`, and forward the rest unchanged â€” bounded memory regardless of response size. `MemoryStore` implements both the minimal `Store` interface and the optional `WaitableStore` extension; `ConflictWait` uses `WaitableStore` when available and falls back cleanly to `ConflictReject` (immediate 409, no polling) for non-waitable stores. `SafeMethods` and `CacheStatuses` both follow nil-vs-empty semantics, allowing GET to participate in idempotency when explicitly requested.

### Notes â€” Phase 10
- `SafeMethods` and `CacheStatuses` (idempotency), and `SafeMethods` (csrf), all use a deliberate `nil` vs `[]T{}` distinction. `nil` means "use built-in defaults"; an empty non-nil slice means "no defaults â€” honor verbatim". This is the only way to opt every method into idempotency or CSRF protection. Three dedicated tests per slice assert each form behaves correctly.
- `idempotency.ConflictWait` requires the configured `Store` to implement `WaitableStore`. Stores that don't are treated identically to `ConflictReject`: they return 409 immediately on contention. There is no polling fallback â€” that would invite busy waits or timing bugs. `MemoryStore` implements both interfaces from day one; future Redis/Postgres stores can opt in by adding a single `Wait(ctx, key)` method without breaking the minimal `Store` contract.
- `ratelimit.New(cfg)` and `idempotency.NewMemoryStore()` start no background goroutines. Cleanup is lazy and in-line. Callers wanting a periodic janitor use `ratelimit.NewWithCleanup(cfg)` or `idempotency.NewMemoryStoreWithJanitor(sweep)` â€” both return a stop function meant to be wired into `app.OnShutdown`. There is no `Plugin()` constructor for either, matching the existing `bodylimit` / `cors` / `secure` / `etag` / `timeout` / `recover` shape.
- `plugins/sanitize` is the only Phase 10 plugin that lives outside the root module. NFC normalization needs `golang.org/x/text/unicode/norm`, and the root module's strict zero-dependency policy disallows that import. The submodule joins the existing `plugins/prometheus` and `plugins/otel` release dance: `replace github.com/nilshah80/aarv => ../..` during development, lifted at tag time.
- Distributed implementations of ratelimit and idempotency (Redis, Postgres) are intentionally **deferred to a later release** (v0.7.x or v0.8.0). The `Store` / `WaitableStore` contract for idempotency, and the (TBD) backend interface for ratelimit, need real production traffic against the in-memory implementations before the contracts can be locked down.

## [0.6.0] - 2026-05-02

### Added
- Core: `(*Context).RoutePattern() string` returns the registered aarv route pattern that matched a request, in path-only form (e.g. `/users/{id}`), or empty for 404, 405, `App.Mount` handlers, and any path outside the registered aarv route table. - Core: `(*Context).SetLogger(*slog.Logger)` swaps the request-scoped logger for the remainder of a request. - verboselog plugin: new `Sink Sink` callback and `SuppressSlog bool` toggle on `Config`.
